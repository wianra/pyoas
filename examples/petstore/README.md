# Petstore Example

A minimal petstore API that shows what pyoas generates from an OpenAPI spec.

## What's in here

```
examples/petstore/
├── openapi.yaml                    # The source spec (10 operations, 3 tags)
├── pyoas.yaml                      # pyoas config (services + tests enabled)
│
├── src/
│   ├── generated/                  # ← Generated by pyoas. Do not edit.
│   │   ├── models/
│   │   │   ├── pets.py             # Pet, PetList, CreatePetRequest, UpdatePetRequest
│   │   │   ├── store.py            # Order, CreateOrderRequest
│   │   │   └── users.py            # User, CreateUserRequest
│   │   └── routers/
│   │       ├── pets.py             # GET/POST /pets, GET/PATCH/DELETE /pets/{petId}
│   │       ├── store.py            # GET /store/inventory, POST/GET /store/orders
│   │       └── users.py            # POST /users, GET /users/{username}
│   │
│   └── services/                   # ← Scaffolded once, safe to edit.
│       ├── pets.py                 # PetsService — in-memory implementation
│       ├── store.py                # StoreService — in-memory implementation
│       └── users.py                # UsersService — in-memory implementation
│
└── tests/
    └── generated/                  # ← Scaffolded once, safe to edit.
        ├── conftest.py             # Pydantic model factories (polyfactory)
        ├── test_pets.py            # Router tests for the Pets tag
        ├── test_store.py           # Router tests for the Store tag
        └── test_users.py           # Router tests for the Users tag
```

## How it works

**Generated files** (`src/generated/`) are fully replaced on every `pyoas generate` run.
They carry the header `# Generated by pyoas — do not edit manually`.

**Scaffolded files** (`src/services/`, `tests/generated/`) are written once and never
overwritten unless you set `services.overwrite: true` or `tests.overwrite: true` in
`pyoas.yaml`. They carry the header `# Scaffolded by pyoas — safe to edit`.

## Regenerate from scratch

Run this from the repo root:

```bash
uv run pyoas generate --config examples/petstore/pyoas.yaml
```

Or from inside the example:

```bash
cd examples/petstore
pyoas generate
```

## Check for drift

This is what CI runs to verify the committed output is up to date:

```bash
uv run pyoas diff --config examples/petstore/pyoas.yaml
# exits 0 if clean, 1 if any file would change
```

## The spec

`openapi.yaml` defines three tags across 10 operations:

| Tag | Operations |
|-----|-----------|
| Pets | `listPets`, `createPet`, `getPet`, `updatePet`, `deletePet` |
| Store | `getInventory`, `createOrder`, `getOrder` |
| Users | `createUser`, `getUser` |

## The services

The `src/services/` files come from `pyoas scaffold services` and have been fleshed out
with an in-memory store to make the example self-contained. In a real project you would
wire in a database session:

```python
# src/services/pets.py
async def get_pets_service(db: AsyncSession = Depends(get_db)) -> PetsService:
    return PetsService(db)
```

## Config explained

```yaml
# pyoas.yaml
spec: openapi.yaml          # path relative to this file

output:
  models: src/generated/models
  routers: src/generated/routers
  source_root: src           # strips "src/" when building import paths

services:
  generate: true
  output: src/services
  import_path: services      # routers import from "services.pets", "services.store", …

tests:
  generate: true
  output: tests/generated
```
